    Prometheus Federation - Detailed Implementation Guide #

    1. Introduction #

    Prometheus Federation is a technique that allows one Prometheus server (global/federated Prometheus) to scrape metrics from one or more other Prometheus instances (child Prometheus) via their /federate endpoint.

    Key Goals #

    • Centralize metrics from multiple sources (multi-cluster, multi-datacenter).
    • Aggregate precomputed metrics (recording rules) to reduce query load.
    • Maintain local alerting at each cluster to avoid dependency on network connectivity to the central server.

    2. Architecture Overview #

    3. Federation Concepts #

    3.1. Child Prometheus #

    • Scrapes metrics locally from exporters, Kubernetes services, etc.
    • Can run alerting rules locally to ensure cluster-level alerts are not dependent on the central instance.
    • Publishes selected metrics through the /federate endpoint.

    3.2. Global/Federated Prometheus #

    • Does not scrape individual exporters directly.
    • Pulls aggregated or selected metrics from child Prometheus instances.
    • Ideal for global dashboards and cross-cluster SLO/SLA monitoring.

    4. Federation Use Cases #

    • Multi-Cluster Kubernetes Monitoring

      • Each cluster runs its own Prometheus stack for reliability.
      • A central Prometheus aggregates selected metrics for a unified view.

    • Cross-Region or Multi-DC Infrastructure

      • Regional Prometheus instances monitor local infrastructure.
      • A central instance combines them for executive reporting and long-term analysis.

    • Security/Network Isolation

      • Child Prometheus stays within the secured environment.
      • Global Prometheus only needs minimal network access to /federate.

    5. Federation Flow #

    1. Child Prometheus scrapes metrics from exporters.
    2. It applies recording rules to generate summarized or precomputed metrics.
    3. The Global Prometheus scrapes /federate endpoints from each child, selecting only needed metrics via match[] parameters.
    4. Global Prometheus stores these metrics with external labels (e.g., cluster, region) to identify their origin.

    6. Configuration #

    6.1. Child Prometheus - Add External Labels #

    Add unique identifiers for each Prometheus instance so the global Prometheus can distinguish their data.

    global:
  external_labels:
    cluster: "k8s-a"
    environment: "prod"

### 6.2 Global Prometheus - Scrape Configuration
```yaml
scrape_configs:
  - job_name: 'federate-k8s-a'
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{__name__=~"^(kube_.*|node_.*|cluster_.*)$"}'   # Select only required metrics
    static_configs:
      - targets:
          - 'prometheus.k8s-a.svc.cluster.local:9090'

  - job_name: 'federate-k8s-b'
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{__name__=~"^(kube_.*|node_.*|cluster_.*)$"}'
    static_configs:
      - targets:
          - 'prometheus.k8s-b.svc.cluster.local:9090'

    6.3. Best Practices for match[] #

    Do NOT federate all metrics ({name=~".+"}) unless necessary — this increases load.

    Prefer recording rules in the child Prometheus to precompute:

    Cluster-wide CPU/memory usage

    Application-level request rates

    SLO error budgets

    6.4. Example Recording Rule in Child Prometheus #

    groups:
- name: cluster.rules
  rules:
  - record: cluster:cpu_usage:sum
    expr: sum(rate(container_cpu_usage_seconds_total{job="kubelet"}[5m])) by (cluster)
  - record: cluster:memory_usage:sum
    expr: sum(container_memory_usage_bytes{job="kubelet"}) by (cluster)

    6.5. Example Alerting Rule in Child Prometheus #

    groups:
- name: cluster.alerts
  rules:
  - alert: HighCPUUsage
    expr: cluster:cpu_usage:sum > 80
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "High CPU usage detected"
      description: "CPU usage is above 80% for more than 5 minutes."

    6.6. Example Alerting Rule in Global Prometheus #

    groups:
- name: global.alerts
  rules:
  - alert: HighCPUUsage
    expr: sum(cluster:cpu_usage:sum) by (cluster) > 80
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "High CPU usage detected"
      description: "CPU usage is above 80% for more than 5 minutes."

    6.7. Example Grafana Dashboard Query #

    - expr: sum(cluster:cpu_usage:sum) by (cluster)
  legendFormat: '{{cluster}}'
  refId: A

    7. Summary #

    Prometheus Federation is a powerful technique for aggregating metrics across multiple Prometheus instances, enabling centralized monitoring and alerting. By carefully configuring child and global Prometheus instances, organizations can achieve a scalable and efficient monitoring solution that meets their needs.
